Field Validation & Testing

Table Structure

Each table has the following fields and types, I got this by running the following SQL query on the trip_raw_jan_2022 table:

SELECT * FROM my-project-1-359120.cyclistic_trip_data.INFORMATION_SCHEMA.TABLES
where table_name = "trip_raw_jan_2022";

The data is organised as 13 columns where each row (observation) represents one bike transaction.

A transaction is when a rider takes a bike from the start station and returns it to an end station. This transaction generates data which populates one row in a table.

A unique ride_id is generated per transaction ("hire") along with the following self-explanatory fields:

-- Field Names
ride_id STRING,
rideable_type STRING,
started_at TIMESTAMP,
ended_at TIMESTAMP,
start_station_name STRING,
start_station_id STRING,
end_station_name STRING,
end_station_id STRING,
start_lat FLOAT64,
start_lng FLOAT64,
end_lat FLOAT64,
end_lng FLOAT64,
member_casual STRING

Field: ride_id (STRING)

The ride_id is the primary key of the database. Each time a customer uses a bike a unique ride_id is created. Any records containing duplicate ride_ids should be removed. To test for duplicates we use the following query:

select ride_id, count(ride_id)
from my-project-1-359120.cyclistic_trip_data.trip_raw_2022
group by ride_id
having count(ride_id) > 1

Result : No results Comments: There are no duplicate records present, which is good for data integrity of the dataset.

Business task: How do annual members and causal riders use Cyclistic bikes differently?

We can then omit the need for the names of the start and end stations and their ids as we have the GPS data which can be plotted in Tableau.
